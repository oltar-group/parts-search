# Вступ {#вступ}

**Мета документа:** Метою цього документа є забезпечення зрозумілого, детального та структурованого керівництва для інтеграції зовнішніх систем із публічним API. Документ надає опис процесів, які можна виконувати за допомогою методів API, такі як отримання списків товарів, оформлення замовлень, оновлення даних тощо.

Цільова аудиторія: розробники, інтегратори та технічні спеціалісти, які працюють над впровадженням чи інтеграцією системи, використовуючи публічне API.

Документація публічного API версії 1.0 компанії AUTONOVA-D (закритий бета-тест для обмеженої кількості компаній). Ми будемо раді отримати від компаній, що використовують це API, зворотний зв'язок із зауваженнями щодо його використання та побажаннями щодо покращення функціональності. Прохання надсилати зворотний зв'язок співробітнику компанії, який здійснює пряму комунікацію з представниками вашої компанії.

**Основні процеси які може виконувати API:**

Скориставшись інструментами публічного API, ви зможете:

1. Завантажувати у вашу систему дані щодо наявності товару.  
2. Завантажувати у вашу систему актуальні цінові пропозиції.  
3. Сформувати замовлення.   
4. Отримувати дані по ваших замовленнях.  
5. Отримувати дані щодо способів доставки, складів та філій нашої компанії.

# Авторизація користувача  {#авторизація-користувача}

Існує два середовища для інтеграції:

Тестове середовище:  
***https://api.autonovad.ua/dev***

Використовується для впровадження та відлагодження. Саме тут потрібно реалізовувати створення замовлень, тестувати авторизацію, запити тощо.

Робоче середовище:  
[***https://api.autonovad.ua/stable***](https://api.autonovad.ua/stable)

Створення замовлень дозволяється лише після впровадження на тестовому середовищі.  
GET-запити дозволено використовувати на будь-якому етапі для отримання актуальних даних.

Для початку роботи з інтеграцією через публічне API клієнту необхідно виконати наступні кроки:

1. **Подання запиту на доступ**  
   Клієнт/партнер компанії, який бажає провести інтеграцію, повинен звернутися до відповідального співробітника компанії із запитом. У запиті мають бути вказані:  
   * Назва компанії/організації.  
     Код клієнта з сайту компанії.  
   * Електронна пошта для реєстрації.  
   * Мета використання API.  
2. **Авторизація користувача**  
   Після затвердження та створення облікового запису користувач отримує дані для авторизацію. Дані для авторизації представляють під собою логін та пароль, який використовується для отримання токена.  
3. **Отримання токена**  
   Для авторизації та отримання тимчасового токена потрібно використати метод:

POST/api/v1/auth/token 

У відповідь на авторизаційний запит клієнт отримає JSON-об’єкт із токеном доступу (access\_token), який використовується для всіх наступних запитів до API.

Даний токен тимчасовий та діє протягом 300 секунд, після закінчення даного часу потрібно повторно отримати токен або оновити токен за допомогою методу:

GET/api/v1/auth/token/refresh/{refreshToken} 

В подальшому в багатьох запитах буде фігурувати такий параметр **clientId** – це код клієнта, який доступний на сайті, та який вказувався при реєстрації клієнта для отримання даних для авторизації та отримання токена. Даний код так само як і логін і пароль для авторизації є незмінним, його не можна отримати шляхом використання методу.  

# Отримання інформації про товари через API {#отримання-інформації-про-товари-через-api}

На даний момент отримати актуальну інформацію про залишки товарів та ціни можна кількома способами:

1. Перевірити наявність позицію на сайті компанії.  
2. Уточнити інформацію про наявність позиції у менеджера по продажам. 

Для автоматизації процесу пошуку позиції існують наступні методи:

GET/api/v1/wares/producers

Цей метод повертає список виробників, доступних у системі. Отримані дані можна використовувати для пошуку запчастин, звужуючи результати до продукції конкретного виробника.

GET/api/v1/wares/clients/{clientId}/parts/{partId}

Цей метод використовується для отримання всієї доступної інформації про конкретний продукт (запчастину) для вказаного клієнта. Він дозволяє отримати детальні дані, необхідні для аналізу або замовлення товару, включаючи характеристики, доступність, ціни та іншу пов’язану інформацію.

При використанні методу потрібно звертати увагу на заповнення параметра **FilterByResultCategory,** він може мати 3 значення:

* 1 – клієнт отримує масив даних про пропозиції що знаходяться на його філії, тобто якщо клієнт у Києві він отримує дані про наявність товару на київській філії.   
* 2 – клієнт отримує масив даних про пропозиції що знаходяться на інших філіях компанії. Тобто якщо в Одесі запчастини немає, клієнт може за допомогою даного методу отримати дані про всі деталі на інших філіях і вирішити звідки він планує замовити.   
* 3 – клієнт отримує масив даних про пропозиції що знаходяться в постачальників компанії, тобто дану запчастину можна буде поставити під замовлення.

* 1,2,3 \- отримуємо всю інформацію по наявності товару. 

Даний параметр потрібно вводити у формат **/api/v1/wares/clients/{clientId}/parts/{partId}?FilterByResultCategory=3**

Метод повертає **масив результатів**, а не одну позицію — це очікувано. Якщо потрібно точне співпадіння, використовуйте GET /article/{artId}/brands/{brandId}/parts/{partId}.

Якщо запчастини немає — повертаються **аналоги**

GET/api/v1/wares/article/{articleId}

Цей метод використовується для отримання інформації про товари за їхнім унікальним каталожний номером, який присвоєно виробником. Основна його задача — швидкий доступ до детальної інформації про товари з однаковим каталожний номером. Тобто за допомогою даного метода можна отримати дані про всі запчастини, наявні в компанії, з однаковими каталожний номером, але від різних виробників. 

Артикул (**articleId**) не є унікальним — він може бути спільним у кількох брендів. Щоб отримати точний результат, використовуйте також **brandId** або **partId**.

GET/api/v1/wares/article/{artId}/brands/{brandId}/parts/{partId}

Цей метод використовується для отримання інформації про товари для здійснення точного пошуку та отримання інформації про товарну картку за такими параметрами:

| Назва | Параметр | Визначення |
| :---- | :---- | :---- |
| Каталожний номер товару | **artId** | Унікальний код, присвоєний виробником |
| Виробник  | **brandId** | Ідентифікатор бренду, до якого належить товар |
| Номер запчастини | **partId** | Унікальний ідентифікатор конкретної запчастини або моделі |

Цей метод дозволяє знайти і отримати повну інформацію **про одну єдину картку товару**, яка відповідає вказаним критеріям. Що може використовуватися для:

* Точного пошуку: Виключає випадкові збіги, повертаючи лише один товар, що відповідає всім параметрам.  
* Операцій з конкретним товаром: Наприклад, перегляд детальної інформації перед купівлею. 

Якщо в параметр ввести значення 0, це буде розцінено як використання методу без урахування цього параметра.

GET/api/v1/warehouses

Цей метод використовується для отримання інформації про всі склади компанії. Отримані дані можуть бути використані для визначення місця знаходження певної запчастини, а саме на якому зі складів вона зберігається. Ця інформація є корисною для планування логістики, управління запасами та забезпечення оперативного обслуговування клієнтів.

GET/api/v1/affiliates

Цей метод надає інформацію про всі філіали компанії. Отримані дані можуть бути використані для визначення філіалу, де зберігається певна запчастина, на основі даних про склад зберігання. Ця інформація дозволяє ефективно координувати логістичні процеси та забезпечувати швидкий доступ до потрібних запчастин.

# Отримання цінові пропозиції через API {#отримання-цінові-пропозиції-через-api}

На даний момент отримати актуальну інформацію про ціни можна кількома способами:

* Через сайт компанії, знаходячи потрібну позицію вручну.  
* Отримавши прайс-лист, який надсилається на пошту щоранку.

Однак, це також можливо автоматизувати за допомогою API. Використовуючи спеціальні методи API, можна:

GET/api/v1/prices/clients/{clientId}

Цей метод використовується для отримання масиву прайсів (назва прайсу, ID прайсу, номер прайсу), доступних конкретному клієнту відповідно до його унікального ідентифікатора. 

Система прайсів працює таким чином, що клієнт сам обирає позиції, які його цікавлять, і хоче отримувати оновлення цін саме по них. Це виключає необхідність отримувати весь масив товарів компанії з цінами. Такий підхід забезпечує персоналізований доступ до інформації, підвищує зручність використання та оптимізує процес взаємодії з даними. Запит на підключення розсилки та подальшу можливість отримання їх через API можна звернувшись до представника компанії.

GET/api/v1/prices/{priceId}

Цей метод використовується для отримання файлу з актуальним прайсом на основі даних, отриманих із попереднього методу. Тобто вказавши ID прайсу, який цікавить, та формат xlsx або xls в параметри запиту метод поверне файл відповідного прайсу.

Ці методи дозволяють швидко і зручно інтегрувати дані про товари та ціни без необхідності вручну переглядати сайт або очікувати ранковий прайс-лист.

GET/api/v1/wares/clients/{clientId}/stock

Цей API-метод використовується для перевірки наявності товару на складах перед оформленням замовлення. Він дозволяє отримати інформацію про доступні запчастини чи моделі на конкретному складі, що допомагає уникнути нестачі товару та оптимізувати процес закупівлі.

Метод повертає всі доступні позиції, включно з аналогами. Якщо потрібно перевірити наявність конкретної позиції — вкажіть точний **partId** та **warehouseId**.

| Назва | Параметр | Визначення |
| :---- | :---- | :---- |
| ID клієнта | **clientId** | Код клієнта. Параметр є обов’язковим. |
| Код запчастини (**partId)** | **spareCodes** | Унікальний ідентифікатор конкретної запчастини або моделі. Параметр є обов’язковим. |
| Номер складу | **warehouseId** | Унікальний ідентифікатор складу. |

# Отримання адрес та перевізників через API {#отримання-адрес-та-перевізників-через-api}

Методи, описані в цьому розділі, призначені для отримання даних, необхідних для подальшого створення замовлень через API. Зокрема, вони дозволяють вносити інформацію про доставку для цих замовлень. Дані методи дублюють процес введення даних про доставку, який зазвичай виконується вручну на сайті під час створення замовлення.

GET/api/v1/addresses/regions

Цей метод використовується для отримання масиву регіонів/областей та їхніх кодів із системи компанії. Отримані дані можуть бути використані в інших методах як параметри для передачі інформації, пов'язаної з регіонами, наприклад для отримання міст в області. 

GET/api/v1/addresses/regions/{regionId}/cities

Цей метод використовується для отримання масиву міст та їхніх кодів із системи компанії відповідно до зазначеного в параметрах коду регіону (**regionId**), також даний метод виводить значення (**NpId**), що можна використати при формуванні замовлення з доставкою через поштовою службою **Нова Пошта**. Отримані дані можуть бути використані в інших методах, наприклад для отримання перевізників доступних клієнту в певному місці.

GET/api/v1/addresses/carriers/clients/{clientId}/city/{cityId}

Цей метод використовується для отримання списку перевізників, доступних конкретному клієнту в зазначеному місці. Для виконання запиту необхідно вказати в параметрах код клієнта (**clientId**) та код міста (**cityId**). Отримані дані дозволяють визначити доступні варіанти перевізників для подальшого оформлення замовлень або вибору способу доставки.

GET/api/v1/addresses/delivery-points/city/{cityId}/carrier/{carrierId}

Цей метод використовується для отримання списку точок доставки у конкретному місті від зазначеного перевізника (Нова Пошта, Міст Експрес і т.д.). Для виконання запиту необхідно вказати в параметрах код міста (**cityId**) та код перевізника (**carrierId**). Отримані дані дозволяють клієнту вибрати відповідну точку доставки для оформлення замовлення.

GET/api/v1/addresses/clients/{clientId}

Цей метод виводить всі адреси доставки, внесені в систему для конкретного клієнта (**clientId**). Отримані адреси можуть бути використані для створення замовлення через API при доставці товару клієнту транспортом компанії. Дані адреси зазвичай створюються вручну клієнтом на сайті та зберігаються в системі для подальшого використання під час оформлення замовлення.

GET/api/v1/deliveries/clients/{clientId}/address/{addressId}

Цей метод використовується для отримання доступних часових проміжків, у межах яких можлива доставка замовлення клієнту за вказаною адресою. Для виконання запиту необхідно зазначити код клієнта (**clientId**) та код адреси (**addressId**). Отримані дані дозволяють планувати доставку замовлень відповідно до зручного для клієнта часу.

GET/api/v1/deliveries/clients/{clientId}/address/{addressId}

Попередній метод, але оновлений відповідно до потреб клієнтів. Цей метод використовується для отримання доступних часових проміжків, у межах яких можлива доставка замовлення клієнту за вказаною адресою. Для виконання запиту необхідно зазначити код клієнта (**clientId**) та код адреси (**addressId**). Отримані дані дозволяють планувати доставку замовлень відповідно до зручного для клієнта часу.

	У відповіді якого ви отримаєте:

* **StartDeliveryTime** — час початку доставки  
* **OrderUntil** — час, до якого можна оформити замовлення (береться з налаштувань перевізника)  
* **Available** — чи можна використати цей часовий проміжок для створення замовлення прямо зараз

# Формування замовлення та робота з ним через API {#формування-замовлення-та-робота-з-ним-через-api}

У цьому розділі буде описано методи API, за допомогою яких можна створювати замовлення та отримувати всю необхідну інформацію по них, аналогічно до того, як це відбувається на сайті компанії. Ці методи дозволяють інтегрувати процес оформлення замовлень у ваші системи та автоматизувати його.

Раніше описані методи API використовувалися для отримання необхідної інформації, такої як прайси, товари та їх доступність, яка є основою для формування замовлення. Використовуючи дані, отримані з цих методів, ви можете легко створювати замовлення через API, забезпечуючи швидкість і точність у взаємодії з системою компанії.

POST/api/v1/orders 

Цей метод є ключовим у даному API. Його основна функція — створення замовлення. Більшість методів, описаних у попередніх розділах, безпосередньо або опосередковано використовуються для підготовки даних, необхідних для формування тіла запиту цього методу, або для подальшого отримання інформації, пов'язаної із замовленнями.

Тобто даний метод забезпечує можливість створення замовлень через API аналогічно до процесу оформлення на сайті компанії, використовуючи всі отримані раніше дані (товари, прайси, клієнти, тощо).

Як це працює:

* **Підготовка даних**:

Методи для отримання прайсів, списків товарів, варіантів поставки і т.д. надають всю необхідну інформацію для побудови коректного запиту.

* **Створення замовлення**:

Використовуючи зібрані дані, формується тіло запиту для методу **POST/api/v1/orders**, яке містить деталі замовлення, такі як список товарів, кількість, обрані ціни та інформацію про клієнта.

Прошу зауважити що дані параметри не є обов’язковими для заповнення:

  **"AuthorId": "999999"**  
  **"AuthorName": "Ivanov Ivan"**

* **Подальша робота із замовленням**:

Після успішного виконання методу користувач отримує відповідь та код даного замовлення.

Цей метод є центральним елементом системи API, оскільки саме він дозволяє завершити процес, початий через інші методи, і створити замовлення, що відповідає реальним бізнес-процесам компанії.

**Загальні вимоги (FAQ)**

* Обов’язкові поля: ClientId, Positions.CartPosition\[\], Delivery.DeliveryTypeId.  
* Поле Date не є обов’язковим. Якщо не передане — буде встановлене автоматично на сервері.  
* Поле ClientPrice у кожній позиції обов’язкове — ціна має бути зафіксована під час створення.

**CartPosition**

* Id — службове поле, передавайте 0 або не вказуйте.  
* WareId — обов’язкове, унікальний ідентифікатор товару.  
* SaleQnt — необов’язкове, кратність. Передавайте 0, якщо не застосовується.  
* SupplierId, SupplierUid — ID постачальника, беруться з методу /wares/clients/{clientId}/parts/{partId}. Можуть збігатися.  
* WareQnt — обов’язкова кількість товару.  
* WarehouseId — необов’язкове. Якщо не передане, склад визначається автоматично або через Delivery.WarehouseId.

**Options.OrderOptions**

* SendFullOrder — якщо true, замовлення буде створене лише за умови наявності всіх позицій.  
* SendOrderWithoutDocuments — дозволяє оформити без супровідних документів.  
* NeedToCheck — відмітка, що замовлення потребує перевірки менеджером.

**Delivery.DeliveryTypeId**

Тип доставки, визначається відповідно до перевізника.

(/api/v1/addresses/clients/{clientId}) у відповіді є TransporterServiceId \- це код перевізника

В методі по отриманню перевізників (/api/v1/addresses/carriers/clients/{clientId}/city/{cityId}) є "DeliveryTypeId" який вам і потрібен

**DeliveryPayload (для типів 1 та 4\)**

* AddressId — ID адреси доставки, з методу /addresses/clients/{clientId}.  
* DeliveryMethodId — ID перевізника з /addresses/carriers/clients/{clientId}/city/{cityId}.  
* DeliveryMethodName — НЕ потрібно заповнювати. Заповнюється автоматично й повертається у відповіді на /orders/{orderId}/clients/{clientId}.  
* TransporterServiceId — ID служби доставки (може бути в тому ж методі, що й перевізник).  
* DeliveryTime — обирається з методу /deliveries/clients/{clientId}/address/{addressId} (поле MaxDeliveryTime). Замовлення має бути створене до MinOrderTime.  
* Insurance — опціонально.  
* PickupPointId — не передавати, якщо це не самовивіз.

**PickUpPayload (для типу 2 — самовивіз)**

* PickupPointId — ID точки видачі з /addresses/delivery-points/....  
* PickupTime — НЕ заповнювати. Визначається компанією.  
* CollectUrgently — опціонально.

Якщо тип доставки ≠ 2, блок PickUpPayload лишається у тілі, але всі поля порожні або не передаються взагалі.

**Додатково**

* Для отримання WareId, SupplierId, ProducerId — використовуйте метод /wares/clients/{clientId}/parts/{partId}.  
* Для вибору складу вручну використовуйте Delivery.WarehouseId. Якщо не вказано — буде обрано автоматично.  
* Якщо API повертає 400 — перевірте обов’язкові поля, кількість товару \> 0, структуру JSON.  
* Якщо API повертає 403 — перевірте правильність токена, clientId та прав доступу до ресурсів (адреси, склади).  
* Створення тестового замовлення без реєстрації не передбачене. Використовуйте середовище /dev.  
* Якщо працюєте з кількома акаунтами клієнта — для кожного потрібен окремий логін і токен.

GET/api/v1/orders/{orderId}/clients/{clientId}

Цей метод використовується для отримання інформації про конкретне замовлення клієнта. Дані можуть бути надані як з деталями про товари, так і без них. Для виклику методу необхідно вказати номер замовлення (**orderId**) та номер клієнта (**clientId**) та чи потрібно виводити дані про товар чи ні. Отримана інформація дозволяє переглядати деталі замовлення, включаючи дані про товари.

GET/api/v1/orders/statuses

Цей метод повертає масив усіх можливих статусів, які може мати замовлення під час його обробки та виконання. Кожен статус містить інформацію про те, чи є він фінальним. Отримані дані дозволяють відстежувати етапи обробки замовлення та визначати його поточний стан.

GET/api/v1/orders/statuses/positions

Цей метод аналогічний до методу попереднього, але повертає масив статусів для позицій у замовленні, а не для всього замовлення в цілому. Кожен статус також містить інформацію про те, чи є він фінальним. 

GET/api/v1/orders/client/{clientId}

Цей метод використовується для отримання інформації про замовлення клієнта. Для виконання запиту необхідно вказати ідентифікатор клієнта (clientId). Отримані дані включають деталі про всі замовлення клієнта, що дозволяє аналізувати історію замовлень, їхній статус та вміст.

GET/api/v1/orders/positions/client/{clientId}

Цей метод аналогічний до методу отримання даних про замовлення клієнта, але повертає детальну інформацію про окремі позиції у замовленнях клієнта. Для виконання запиту необхідно вказати ідентифікатор клієнта (**clientId**). Отримані дані дозволяють аналізувати стан, характеристики та деталі кожної позиції з усіх замовлень клієнта.

# Отримання видаткових документів через API {#отримання-видаткових-документів-через-api}

API надає два методи для отримання інформації про документи відвантаження, пов’язані з оформленими замовленнями:

GET/api/v1/documents/shipments

Цей метод використовується для отримання списку видаткових документів (накладних) за період.

**Пояснення ключових полів:**

* orderUuid — ідентифікатор замовлення;  
  shipmentDocs — масив документів відвантаження:  
  * docAmount — сума документу;  
  * docDate — дата створення;  
  * docNumber — номер документа;  
  * positions — перелік позицій у документі з цінами, кількістю та даними по виробнику.

GET/api/v1/documents/shipments/orders/{orderUuid}

Цей метод використовується для отримання інформації по видатковим накладним згідно коду замовлення. 

Повертає список видаткових документів з повною деталізацією по позиціях.

**Пояснення ключових полів:**

* артикул (wareArticle),  
* виробник (wareManufacturer),  
* кількість (quantity),  
* ціна (warePrice),  
  сума (amount).
